# 📋 01 — Requirements & User Flows
---

## 📝 1. Обзор

> Приложение, для решения проблем с нехваткой времени и организованностью.
> Интерактивный менеджер задач для группы с простым интерфейсом позволит удобно
> создавать задания, видить их прогресс, статистику, требования и вклад каждого
> участника. Нацелено на работу с музыкой.

## 👥 2. Роли пользователей

| Роль | Описание | Права |
|---|---|---|
| Member | Обычный участник команды, зарегистрированный на сайте | Создаёт, редактирует и удаляет **любые** задачи и подзадачи (не только свои), видит задачи всех |

>💡 На данный момент разделения на роли нет, у всех участников одинаковые права.

> ⚠️ **Открытая регистрация:** создать аккаунт может любой незарегистрированный и незалогиненный посетитель, без инвайта. Сознательный отход от исходной идеи "приватная команда по инвайтам". Практическое следствие: у приложения нет барьера входа. Если это станет проблемой — потребуется invite-token или роль admin, одобряющая новых участников. Открытый вопрос, см. раздел 7.

## ⚙️ 3. Функциональные требования

### ✅ 3.1 Задачи (Tasks)

- Пользователь может создать задачу с полями:
  - 🏷️ **название**
  - 📝 **описание** (необязательно)
  - 📅 **конечный срок выполнения (DueDate)**
  - 📅 **дату создания (CreatedAt)**
  - 👤👥 **исполнитель/и**
  - 🚦 **статус** (подробнее ниже)
- 📂 Задача может содержать **подзадачи**
- 🗓️ Задача отображается в календаре на отметке текущего времени и плавно перемещается относительно его
- ⏰ Просроченные задачи (дошедшие до DueDate срока, но не выполненные) показываются отдельно от сегодняшних

### 📂 3.2 Подзадачи (Subtasks)

- 🔗 Подзадача принадлежит **одной задаче**, срочность (DueDate) наследует от неё — своего DueDate не имеет
- Например:
  - 🎵 Задача - сделать трек
  - 🎼 Подзадачи
    - 🥁 написать бит
    - ✍️ придумать текст
    - 🎤 записаться
    - 🎚️ свести
- Пользователь может создать подзадачу с полями:
  - 🏷️ название
  - 🚦 **статус**
  - 👤👥 **исполнитель/и**

### 3.3 Статусы

- ⏳ Для выполнения (ToDo)
- 🔄 В процессе (InProgress)
- ✅ Готово (Done)
- ❌ Просрочено (Overdue)

> ⚙️ **Overdue — вычисляемое состояние, не хранимое поле.** В БД `status` (у Task и Subtask) хранит только 3 значения: `ToDo`, `InProgress`, `Done`. "Просрочено" — это не отдельное значение статуса, а производный флаг, вычисляемый в момент запроса: `dueDate < сейчас AND status != Done` (у Subtask — по `dueDate` родительской Task, так как своего DueDate у неё нет).
>
> Осознанно отказались от хранения статуса `Overdue` через периодическую фоновую джобу — вариант рассматривался, но даёт either задержку между тиками джобы (статус в БД временно "врёт"), либо требует точного поштучного планирования на каждую задачу, что оверинжиниринг для этого масштаба проекта. Вычисление на лету даёт всегда точный результат без единой лишней строчки инфраструктуры.
>
> Единственное, для чего в проекте всё же нужен фоновый `IHostedService` — email/in-app уведомление "задача просрочена" (это push, не может быть вычислено по запросу). Для этого не нужен статус в БД — достаточно метки времени последней отправки уведомления по задаче, чтобы не дублировать письма.

### 📦 3.4 Выполненные задачи
- ✅ После завершения задача переносится в группу выполненных.
- 👤 Каждый пользователь, указанный исполнителем, получает "+" в статистику выполненных задач, когда задача становится выполненной
- 👥 Каждый пользователь, указанный исполнителем, получает "+" в статистику выполненных подзадач, когда подзадача становится выполненной

### 3.5 Невыполненные задачи
- ❌ Просроченные задачи (см. 3.3 — вычисляемые) показываются отдельно.
- 👥 Не успевший выполнить задачу/подзадачу исполнитель получает "-" в статистику выполненных подзадач (тоже вычисляется агрегатным запросом, не хранится)
- 🔄 У пользователя есть возможность пересоздать невыполненную задачу. Новая задача сохраняет данные предыдущей (кроме `dueDate`, который задаётся заново), но допускает моментальное изменение отдельных полей в том же действии. Новая задача хранит ссылку на исходную (`recreatedFromTaskId`).

### 🔐 3.6 Аутентификация

- Приложение содержит простую аутентификацию (логин/пароль). Всю работу с задачами могут осуществлять только залогиненные пользователи.
- Регистрация открыта для любого незалогиненного посетителя (см. предупреждение в разделе 2).

## 🔄 4. User Flows

### 🔐 Flow 1 - Регистрация/Вход
1. 🛂 Пользователь регистрируется/входит в приложение по логину и паролю

### ✨ Flow 2 — Создание задачи
1. 📄 Открыть страницу
2. ➕ Нажать «Создать задание»
3. ✍️ Заполнить поля
4. 💾 Сохранить
5. 📋 Увидеть задачу в списке

### 🔨 Flow 3 — Выполнение задачи
1. ✅ Отметить подзадачу выполненной
2. 🗓️ С течением времени положение задачи меняется

### 🏁 Flow 4 — Завершение задачи
1. 📦 Выполненная задача переносится в архив выполненных.
2. ❌ Невыполненная — в список просроченных (вычисляется, никуда физически не "переносится" в БД).

### 🔄 Flow 5 — Пересоздание просроченной задачи
1. ❌ Пользователь видит просроченную задачу в архиве
2. 🔁 Нажимает «Пересоздать»
3. ✍️ Опционально меняет часть полей (обязательно указывает новый `dueDate`)
4. 💾 Сохраняет — создаётся новая задача, старая остаётся в архиве, помечена как источник (`recreatedFromTaskId` на новой указывает на неё)

## 📏 5. Бизнес-правила и ограничения

- ✏️ Может ли пользователь редактировать чужую задачу?
  - **Да.** Ограничения "редактирует только создатель" нет — любой залогиненный участник может редактировать и удалять любую задачу и подзадачу.
- 🗑️ Можно ли удалить задачу, если у неё есть подзадачи?
  - Да, подзадачи удаляются каскадно.
- 👥 Что происходит с задачей, если удалили исполнителя?
  - Исполнителей можно удалять до тех пор, пока не останется один. Задача не может остаться без исполнителей.
- 📅 Может ли задача быть без конечной даты?
  - На данном этапе нет.
- 🔁 Кто может пересоздать просроченную задачу?
  - Любой залогиненный пользователь.
